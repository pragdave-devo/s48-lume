> **Tip:** Mermaid is great for quick diagrams, but move away for a predefined layout, and you'll be
>    fighting to get the layout you want. picjs is a new JavaScript library that generates consistent,
>    predictable drawings using constraint-based layout. In this post, we'll introduce picjs and show
>    how it can enhance your documentation and presentations.


We generate a lot of diagrams at [Strike48 Labs](https://labs.strike48.com).
A lot of diagrams. And of course we strongly prefer them to be written in text, embedded in our
docs, and stored in our repos.

Mermaid is a great tool for this, and we use it a lot. But Mermaid's layout engine is pretty
limited. It does a good job of laying out simple diagrams, but as soon as you want to do something
a little more complex, you find yourself fighting with the layout engine to get the diagram to look
the way you want.

I was fighting it one day, trying to create an architecture summary, when I thought back to a tool I
used to use many, many years ago. `pic` was part of the Designer's Workbench (DWB) from Bell Labs.
It was a preprocessor for their `troff` typesetting package, written to generate all their
documentation.

`pic` is an elegant little language. You could embed it in the body of a document, and a
preprocessor would replace it with low-level drawing instructions.

```
.PS
circle "Input"
arrow
box "Process"
arrow
circle "Output"
.PE
```

I went hunting for modern versions. [`dpic`](https://gitlab.com/aplevich/dpic) is an impressive
implementation, but probably over-the-top for a tool I wanted to run in my browser.
[pikchr](https://pikchr.org/home/doc/trunk/homepage.md) is a considerably simpler implementation,
but still a full `pic` equivalent. Only downside: it is written in C, and it uses a bespoke parser
generator.

So I pointed Claude Code at `pikchr` and asked it for a TypeScript version. An hour or so later, I
was looking at diagrams in my browser. Of course, the curse of AI is that it's easy to ask for just
one more wafer thin feature, but at the end of the day I had something that worked like Mermaid,
reading diagram descriptions from HTML pages and rendering them as SVG. It's called `picjs`.


# picjs: when Mermaid won't do what you want…

> **Tip:** `picjs` let's you define tricky diagrams with text. You control the
>   positioning using constraints, so your results are both beautiful and
>   predictable.

 Let's play with it a little.

By default, objects are positioned to the right of the previous object.

``` picjs example
"Input"
arrow
box "Process"
arrow
"Output"
```

You can change the direction:

``` picjs example svgwidth="7rem"
down
"Input"
arrow 50%
box "Process"
arrow 50%
"Output"
```

You don't have to lay things out linearly:

``` picjs example
"Input"
arrow
box "Process"
arrow
"Output"

line <-> down from last box.s
box "Memory"

"(json)" small with .n at 1st text.s

```

Notice how you can use cardinal points to indicate the boundaries of objects.

``` picjs example svgwidth="11rem"
B: box wid 1 ht 1
"n"  with .s  at B.n
"ne" with .sw at B.ne
"e"  with .w  at B.e
"se" with .nw at B.se
"s"  with .n  at B.s
"sw" with .ne at B.sw
"w"  with .e  at B.w
"nw" with .se at B.nw
```

In that last example we used `B:` to give the box a label.

Back to our diagram. Let's add a feedback loop:

``` picjs stacked svgwidth="24rem"
"Input"
arrow
box "Process" fill lightblue
arrow
"Output"

line dashed <-> down 70% from last box.s color grey
box "Memory" rad .1 fill lightgreen

"(json)" small with .n at 1st text.s color purple


arc -> from Output.n to Input.n
ellipse "Feedback" at last arc.n fill yellow
```

If `picjs` can't find an explicit label, it looks for an object with the same text.

That arc looks a little weird, so let's change it to a spline so we can add some control points.

``` picjs stacked  svgwidth="24rem"
"Input"
arrow
box "Process" fill lightblue
arrow
"Output"

line dashed <-> down 70% from last box.s color grey
box "Memory" rad .1 fill lightgreen

"(json)" small with .n at 1st text.s color purple


FB: spline -> \
   from Output.n \
   to Output + (-.3,.4) \
   to Output + (-.6,.6) \
   to Input + (.6,.6) \
   to Input + (.3,.4) \
   to Input.n

ellipse "Feedback"  at FB.n fill yellow
```

Still a little funky. Maybe try arcs?

``` picjs  svgwidth="24rem"e stacked
"Input"
arrow
box "Process" fill lightblue
arrow
"Output"

line dashed <-> down 70% from last box.s color grey
box "Memory" rad .1 fill lightgreen

"(json)" small with .n at 1st text.s color purple


ellipse "Feedback"  dashed at Process.n + (0, .5) fill yellow
arc -> dotted from Output.n to Feedback.e
arc -> dotted from Feedback.w to Input.n
```

## Some of the Cooler Stuff

### Blocks

Blocks group shapes together. Those groups can then be laid out just like regular shapes.

``` picjs example svgwidth="80%"
boxht = .2
boxwid = boxht
fill = lightblue
movewid = .2

A: [ down; box "A"; box "B"; box "C" ]
move
B: [ down; box "1"; box "2"; box "3"; box "4" ]
 move
C: [ down; box "X"; box "Y" ]
move
D: [ up; box "6"; box "7"; box "8"; box "9" ]

# draw a box around the last group
box wid D.wid*1.5 ht D.ht*1.2 \
    at D fill yellow \
    behind D
```

Notice we had to tell pcjs to draw the outer box behind the inner boxes. By default, picjs draws things in the order they are defined, so the last thing is on top.

### Interpolation

If you write `ratio< p2, p2 >`, picjs will find the point that is `ratio` of the way from `p1` to `p2`.
It's conventional to write `ratio` as a fraction, so:


``` picjs example
box "A"
move
OP: [
  down
  box "B"
  move 50%
  box "C"
]

arrow from 1/3<A.ne,A.se> to OP.B.w
arrow from 2/3<A.ne,A.se> to OP.C.w

```

Note that we can reference objects inside blocks using dot notation.

### Macros

picjs macros are really dumb, but they're still a time-saver.

A macro is created using `define` followed by a name and a block of picjs code between braces.

``` picjs example
define mybox {
  box wid 1 ht .5 fill lightblue
}

mybox
arrow
mybox
```

When you write the name of a macro, picjs just substitutes in the macro text. One of the cool side
effects of this is that you can add addition parameters to the constructs inside:

``` picjs example
define mybox {
  box wid 1 ht .5 fill lightblue
}

mybox "one" rad .2
arrow
mybox "two" color darkgreen
```

You can pass parameters to a macro by enclosing them in parentheses. Inside the macro body, refer to
them using `$1`, `$2`, etc.


### Variables and Loops

You can assign numeric values to variables with a simple assignment. These can be used to
parameterize your diagram. They are also useful in loops.


``` picjs example svgwidth="14rem"
spin = 3
color_factor = 255/(spin*$pi)

for i from spin*$pi to 1 step .1 do {
  circle at (i*sin(i), i*cos(i)) \
         radius i \
         color oklch(0.4, 0.3, i*color_factor) \
         fill  oklch(0.6, 0.3, i*color_factor)
}
```


### Smoothing Corners

The `radius` attribute can be applied to circles. It also works on boxes, rounding their corners. It
surprisingly also works on lines, smoothing the joins.

``` picjs example
define myline {
  line right 1      \
  then up .5 left 1 \
  then right 1
}

myline
myline radius .2
myline rad .5 ->
```


### Closed Shapes

picjs supports a lot of different shapes, including circles, ellipses, boxes, but what if you want
to make your own? You can outline a shape with lines and then tell picjs to close it. Once the shape
is closed you can fill it and apply a corner radius.

``` picjs example

line right 1 \
then up .5 left .2 \
then down .2 left .2 \
then up .4 left .4 \
then down .7 left .2 close \
fill green rad .2

```

# How picjs Works: A Text-to-Diagram Compiler in TypeScript

To some extent, the original design of the pic language influences not just the picjs language and
the architecture of its implementation. The tight integration of lexing, parsing, and positioning
means that some interesting tradeoffs had to be made.


## The Four-Stage Pipeline

A picjs program flows through four stages: tokenize, parse, layout, render.

``` picjs
define lab {  box invis wid 0 ht .2 $1 }

right
A: [ down
   lab ""
X: box "Source" bold "Text" bold fit fill AliceBlue rad 0.05
]

move 50%

B:[
   down
   lab "Tokenizer"
X: box  "macro" bold "expansion" italic bold fit fill skyblue rad 0.05
]

move same

C: [
   down
   lab "Parser"
X: box "recursive" bold "descent" italic bold fit fill skyblue rad 0.05
]

move same

D:[
   down
   lab "Layout Engine"
X: box "constraint" bold "solver" italic bold fit fill skyblue rad 0.05
]

move same

E:[
   down
   lab "SVG Renderer"
X: box "layered" bold "output" italic bold fit fill skyblue rad 0.05
]

move same

F: [
   down
   lab ""
X: box "SVG" bold fit fill PaleGreen rad 0.05
]

arrow from A.X.e to B.X.w
arrow from B.X.e to C.X.w
arrow from C.X.e to D.X.w
arrow from D.X.e to E.X.w
arrow from E.X.e to F.X.w

```

This is a classic compiler pipeline, but what's unusual is that parsing and
layout are deeply interleaved. Each object is laid out immediately after its
attributes are parsed, because later objects need to know where earlier ones
ended up — "arrow from last box.ne" has to resolve *now*, not in a later pass.


## Macros Live in the Tokenizer

Most languages handle macros as a preprocessing step or in the parser.
picjs does it inside the token stream itself. When the `TokenStream` encounters
an identifier that matches a defined macro, it expands the macro's body
tokens in-place, splicing them into the token array at the current read
position.

This means the parser never sees macros at all; it just sees a stream of
ordinary tokens. Parameter substitution (`$1` through `$9`) happens during
this expansion, and a recursion guard (max depth 10, plus a per-macro `inUse`
flag) prevents infinite loops.

The result is that macros compose naturally. You can define a macro that uses
other macros, pass expressions as arguments, and everything just works because
expansion happens at the token level.


## Recursive Descent Replaces a Parser Generator

The original pikchr uses a Lemon-generated LALR parser (Lemon is the parser
generator behind SQLite). picjs throws that away entirely and replaces it
with a hand-written recursive descent parser.

This is a deliberate trade: LALR parsers are compact and fast, but recursive
descent is dramatically easier to debug, extend, and read. The grammar is
small enough — maybe 20 productions — that the performance difference is
irrelevant, and features like for-loops and assert statements were added
cleanly as new parsing functions.


## Positioning: Compass Points and Constraint Resolution

This is where picjs gets interesting. Every object has nine compass points
(N, NE, E, SE, S, SW, W, NW, C) plus START and END for lines. Positioning
works through constraints rather than absolute coordinates.

``` picjs svgwidth="12rem"
B: box "box" wid 1.2 ht 0.9 fill grey  rad 0.05

dot at B.n;  text "N"  above small at B.n
dot at B.ne; text "NE" above small at B.ne
dot at B.e;  text "E"  small at 0.12 right of B.e
dot at B.se; text "SE" below small at B.se
dot at B.s;  text "S"  below small at B.s
dot at B.sw; text "SW" below small at B.sw
dot at B.w;  text "W"  small at 0.12 left of B.w
dot at B.nw; text "NW" above small at B.nw
```

When you write `box "X" at last circle.ne`, the layout engine:

1. Looks up the circle's northeast offset (computed by the circle's `xOffset`
   method — each shape calculates its own compass geometry)
2. Adds that offset to the circle's center to get an absolute point
3. Determines the box's anchor point (by default, the edge facing the
   incoming direction)
4. Computes the offset from the box's center to its anchor
5. Translates the entire box so its anchor lands on the target

This means you never think about pixel coordinates. You think about
relationships: "this goes to the right of that," "this arrow starts at
that box's east edge." The engine resolves everything to absolute positions
in a single pass.


## The Chop Algorithm

When a line connects two shapes, you usually want it to end at the shape's
boundary, not its center. The `chop` attribute triggers an intersection
calculation, and it's surprisingly geometry-specific.

For **boxes**, picjs uses octant detection. It computes the angle from the
box center to the incoming line, then compares the dx/dy ratio against
tan(22.5°) ≈ 0.414 and tan(67.5°) ≈ 2.414 to determine which of the eight
compass edges the line would cross. The result is the exact point on the
box's edge (accounting for rounded corners).

For **circles**, it's a simple radius projection along the line direction.

For **ellipses**, the x-axis is scaled to make the ellipse circular, the
circle intersection is computed, then the result is scaled back.

``` picjs svgwidth="18rem"
A: circle "A" rad 0.3 fill LightCyan
B: box "B" at 2.0 right of A fill Wheat rad 0.05
arrow from A to B chop "chop" above italic
```


## Text Sizing Without a Font Renderer

picjs runs in environments without font metrics (plain Node.js, browsers
before DOM access). So it carries its own character-width table: 95 entries
covering ASCII 0x20–0x7E, each storing the character's width as a percentage
of the average character width.

When a box has text but no explicit dimensions, picjs walks the string
character by character, sums the widths, applies scaling factors for bold
(×1.1), big/small, and monospace, then calls the shape's `xFit` method to
expand the object to contain the text. It's approximate, but it's consistent
and deterministic — the same input always produces the same layout regardless
of platform.


## Layered Rendering for Z-Order

SVG has no z-index. Elements are painted in document order. picjs solves this
with a multi-pass renderer: it finds the minimum layer number, renders
everything at that layer, then repeats for the next layer up.

The default layer is 1000. When you write `behind OtherObj`, your object's
layer is set to `OtherObj.layer - 1`. This lets you put a filled background
rectangle behind a group of shapes without worrying about declaration order.


## The Y-Axis Flip

picjs uses a mathematical coordinate system internally — Y increases upward,
which is natural for diagramming ("above" means higher Y). But SVG uses screen
coordinates where Y increases downward.

Every coordinate write goes through `pikAppendY()`, which computes:

```
svg_y = (bbox.ne.y - internal_y) * scale
```

This single transform, applied at the SVG output boundary, means the entire
layout engine can work in intuitive math coordinates while producing correct
SVG.


## Wiring Circular Dependencies Without Import Cycles

The module structure has natural circular dependencies: shapes need the
renderer (to emit SVG), the renderer needs layout (for measurement), and
layout needs shapes (to find classes). TypeScript's module system won't
tolerate circular imports.

picjs solves this with a pattern borrowed from C: function pointers set at
initialization time. Each module exports setter functions like
`setPikAppendStyleFn()`. At startup, `picjs.ts` calls these setters to wire
the modules together:

```typescript
setPikValueFn((p, name) => pikValue(p, name, name.length).val);
setPikAppendStyleFn(pikAppendStyle);
setShapeClasses(aClass, sublistClass, noopClass, arcInit);
```

This happens once, lazily, on the first call to `picjs()`. After that, the
function pointers are hot and the modules can call each other freely. It's
not elegant, but it's explicit and it avoids the fragility of circular
import resolution.


## Shapes as Virtual Method Tables

Each of the 14 built-in shapes (box, circle, ellipse, line, arrow, arc,
spline, oval, cylinder, diamond, dot, file, text, move) is defined as a
`PClass` object with a fixed set of method slots:

``` picjs svgwidth="32rem" vtable
define $slot {
  box $1 mono small fit fill $2 rad 0.02 wid 1.5
}

right

[
  down
  $slot("xInit", powderblue)
  move 0.04
  $slot("xNumProp", powderblue)
  move 0.04
  $slot("xCheck", skyblue)
  move 0.04
  $slot("xChop", skyblue)
]

move 0.2

[
  down
  $slot("xOffset", LightGreen)
  move 0.04
  $slot("xFit", LightGreen)
  move 0.04
  $slot("xRender", LightSalmon)
]
```

This is essentially a vtable — the same pattern C++ uses for virtual methods,
but built manually in TypeScript using plain interfaces with function fields.
A null slot means "not applicable" (e.g., lines have no `xChop`; dots have no
`xFit`). The sorted `aClass[]` array makes shape lookup O(log n) by binary
search.

It's a pattern you rarely see in TypeScript, where you'd normally use class
inheritance. But it maps directly from the C original, and it has a practical
advantage: all shape behavior is data, stored in a flat array, easy to
inspect and extend.


## Zero Dependencies, Dual Target

The core library has no runtime dependencies. It ships as both ESM and UMD,
runs in Node.js or the browser, and includes a `processCodeBlocks()` function
for Mermaid-style in-page rendering. The CLI adds only an optional dependency
on `@resvg/resvg-js` for PNG rasterization.

This is possible because the entire pipeline, the tokenizer, parser, layout, and
SVG generation, is pure computation over strings and numbers. No DOM, no
canvas, no font loading. The character-width table means it doesn't even need
font metrics from the environment.
